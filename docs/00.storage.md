# Product Description: Distributed Memory Database Engine

## Overview

This product is a **new database engine built from scratch** for a distributed LLM system: a neural network of neural networks, where independent model nodes communicate over TCP and need a shared, distributed memory substrate.

The database is not a conventional relational database, document store, or vector database. It is a **memory-native database engine** designed to support:

* local development and production deployments
* distributed model-node coordination
* short-term and long-term memory
* embeddings and vector search
* event history and replay
* graph-like associative memory
* provenance, confidence, salience, and decay
* TCP-based distributed operation

The core product concept is a **local-first distributed memory database** that allows each model node to maintain fast local memory while participating in a larger replicated memory fabric.

---

# Product Goals

## Primary Goal

Provide a production-grade database engine that stores, retrieves, ranks, consolidates, replicates, and expires memory for a distributed LLM system.

## Secondary Goals

* Work as a single local database during development.
* Scale into a distributed production cluster.
* Support embedding-native search.
* Support memory lifecycle semantics inspired by human short-term and long-term memory.
* Preserve full provenance and causality of memories.
* Allow model nodes to operate independently during network interruptions.
* Support eventual convergence across distributed nodes.

---

# Non-Goals

The database does **not** need to be:

* a general-purpose SQL database
* a replacement for Postgres/MySQL
* a BI/analytics warehouse
* a generic object store
* a generic vector database only
* a globally serializable distributed transaction system in the first version

Its purpose is narrower: **memory storage and retrieval for distributed AI systems**.

# Core Product Concept

## Memory as the Primary Data Unit

The database stores **memory records** as first-class entities.

A memory record may represent:

* a user interaction
* a model observation
* an inference
* a generated summary
* a fact
* a task state
* a plan
* a correction
* a tool result
* a procedure
* an embedding artifact
* a relationship between concepts
* a distributed-node communication event

Each memory should include:

| Field Category | Purpose                                                                      |
| -------------- | ---------------------------------------------------------------------------- |
| Identity       | Globally unique memory ID                                                    |
| Type           | Working, episodic, semantic, procedural, artifact, summary, correction, etc. |
| Scope          | User, task, session, node, cluster, tenant, or global scope                  |
| Content        | Raw or structured memory payload                                             |
| Embeddings     | One or more vector embeddings                                                |
| Links          | Relationships to other memories                                              |
| Provenance     | Source, producer node, model version, timestamp                              |
| Confidence     | Estimated reliability                                                        |
| Salience       | Importance score                                                             |
| State          | Active, decayed, superseded, contradicted, archived, tombstoned              |
| Time           | Created, updated, accessed, expiration metadata                              |
| Version        | Mutation/consolidation history                                               |

# Memory Model

## Short-Term Memory

Short-term memory is used for active task context and temporary reasoning state.

### Product Requirements

* Low-latency local reads/writes.
* Supports TTL and automatic expiration.
* Supports promotion into long-term memory.
* Can be local to a node, session, or task.
* Does not require global replication before acknowledgment.
* Can be discarded without corrupting durable system state.

### Acceptance Criteria

* A node can write and read short-term memory locally.
* Short-term memories can expire automatically.
* Short-term memories can be queried by node, task, and session.
* Short-term memories can be promoted to long-term memory.
* Short-term memory loss does not compromise durable long-term memory.
* The system can distinguish temporary memory from durable memory.

---

## Long-Term Memory

Long-term memory is durable, queryable, replicated, and suitable for future retrieval.

### Product Requirements

* Stores durable episodic, semantic, procedural, and artifact memories.
* Supports embeddings and metadata.
* Supports graph relationships.
* Supports replayable event history.
* Supports consolidation from short-term memory.
* Supports archival and tombstoning.

### Acceptance Criteria

* Long-term memories survive process restart.
* Long-term memories survive node restart.
* Long-term memories are queryable by ID, metadata, vector similarity, graph relationship, and time.
* Long-term memories preserve provenance.
* Long-term memories can be versioned.
* Long-term memories can be superseded without destructive overwrite.
* Long-term memories can be replicated to other nodes.

---

## Episodic Memory

Episodic memory stores what happened.

Examples:

* user asked a question
* model generated an answer
* node sent a message
* tool returned data
* task started
* task completed
* correction was applied

### Acceptance Criteria

* Events are append-only.
* Events are replayable.
* Events can reconstruct task/session history.
* Events include causal parent references where applicable.
* Events can be queried by session, task, node, and time.
* Duplicate events are handled idempotently.

---

## Semantic Memory

Semantic memory stores what the system believes or has learned.

Examples:

* facts
* summaries
* inferred concepts
* durable user preferences
* domain knowledge
* conclusions from prior work

### Acceptance Criteria

* Semantic memories can be retrieved by vector similarity.
* Semantic memories can be retrieved by metadata filters.
* Semantic memories can link to supporting source memories.
* Semantic memories include confidence and provenance.
* Contradictory semantic memories can coexist.
* Superseded semantic memories remain auditable.

---

## Procedural Memory

Procedural memory stores reusable processes.

Examples:

* task-routing policies
* prompt strategies
* workflows
* reusable plans
* learned behaviors
* model coordination procedures

### Acceptance Criteria

* Procedures are versioned.
* Procedures can be activated, deprecated, or superseded.
* Procedures can reference memories and prior outcomes.
* Procedures can be evaluated against success/failure metrics.
* Procedures can be retrieved by task type, embedding similarity, and metadata.

# Embedding and Vector Search Requirements

The database must treat embeddings as a native storage and query primitive.

## Product Requirements

* Store multiple embeddings per memory.
* Support multiple embedding models.
* Track embedding model name, version, dimensions, and distance metric.
* Support exact vector search for local development and correctness testing.
* Support approximate nearest neighbor search for production.
* Support metadata-filtered vector search.
* Support hybrid search combining vector similarity, keyword/metadata filters, recency, salience, confidence, and graph relationships.
* Support re-embedding when models change.
* Support index rebuilding without data loss.

## Acceptance Criteria

* The database can store embeddings with associated memory records.
* The database rejects invalid vector queries with incompatible dimensions.
* The database supports cosine similarity at minimum.
* The database can query top-K nearest memories.
* The database can filter vector search by memory type, scope, owner node, status, confidence, and timestamp.
* The database can maintain separate indexes for different embedding models.
* The database can rebuild vector indexes from durable memory records.
* The database can return explainable vector search results, including similarity score and ranking factors.

# Event Log and Replay

The database must use an append-only event log as the durable source of truth for mutations.

## Product Requirements

* Every durable mutation is represented as an event.
* Events are append-only.
* Events are checksummed.
* Events are replayable.
* Events support crash recovery.
* Events support replication.
* Events support auditability.
* Events include causal metadata.

## Acceptance Criteria

* The database can recover state from the event log.
* The database detects partial/corrupt event records.
* The database can replay events deterministically.
* The database can rebuild derived indexes from events or canonical memory records.
* Acknowledged durable writes are not lost after a clean fsync policy.
* Event replay is idempotent.
* Events can be streamed to replicas.

# Distributed Operation

The database must support local-first distributed operation over TCP.

## Deployment Modes

| Mode                   | Description                                       |
| ---------------------- | ------------------------------------------------- |
| Embedded local         | Runs inside an application process                |
| Local server           | Runs as a standalone local database process       |
| Single-node production | Durable server without clustering                 |
| Multi-node cluster     | Distributed production deployment                 |
| Edge/local shard       | Node-local memory participating in larger cluster |

## Product Requirements

* Each database node has a durable identity.
* Each node can own memory shards.
* Each node can replicate memory events to peers.
* Each node can continue operating during network interruptions.
* Nodes can rejoin and catch up after disconnect.
* Nodes communicate over a versioned TCP protocol.
* Replication is based on events, not opaque page copying.
* Replication is idempotent.
* Cluster membership is discoverable and inspectable.

## Acceptance Criteria

* A single node can run with no external dependencies.
* Multiple nodes can form a cluster.
* Nodes can join and leave the cluster.
* A disconnected node can continue accepting local writes if policy allows.
* A reconnected node can catch up from peers.
* Replication delay is observable.
* Duplicate replicated events do not corrupt state.
* Conflicting memories are detected and preserved.
* Cluster state can be inspected by operators.

# Consistency Model

The database should support selectable consistency levels.

## Required Consistency Levels

| Level        | Purpose                                     |
| ------------ | ------------------------------------------- |
| Local        | Fast local reads/writes                     |
| Session      | Read-your-writes within a session/task      |
| Causal       | Respect causal dependency chains            |
| Quorum       | Higher confidence distributed reads/writes  |
| Linearizable | Optional for critical metadata, not default |

## Product Requirements

* Local consistency is the default for development and node-local memory.
* Session consistency is available for task workflows.
* Causal consistency is available for derived memories.
* Quorum consistency is available for production-critical records.
* Vector indexes may be eventually consistent.
* Conflict resolution must not silently discard information.

## Acceptance Criteria

* Reads can specify required consistency level.
* Writes can specify durability and replication requirements.
* Causal dependencies are tracked.
* The system can detect concurrent conflicting updates.
* Conflicts are queryable.
* Last-write-wins is not the default for memory records.
* Critical metadata can opt into stronger consistency.

# Summary

The product is a **distributed memory database engine** for AI systems.

Its defining features are:

* memory-native data model
* local-first operation
* production clustering
* short-term and long-term memory
* embedding-native search
* graph-based associative memory
* append-only event history
* causal provenance
* memory lifecycle management
* explainable retrieval
* local development workflow parity with production

The database should be designed around this central requirement:

> A distributed LLM node must be able to remember, retrieve, forget, consolidate, explain, and replicate knowledge across a network while still working locally and independently.
